### Post 16: Security Architecture
**Format:** Long-form / Security focus

> Security was a first-class concern from day one of building Keevan Store:
>
> **Authentication & Authorization:**
> - Supabase SSR with cookie-based sessions
> - Role-based access (admin, creator, buyer)
> - Row-Level Security on every database table
>
> **Payment Security:**
> - Three-way payment verification (client redirect → IPN webhook → manual API)
> - Atomic payment finalization with database-level locking
> - CSRF protection on all state-changing requests
>
> **File Security:**
> - Magic-byte validation (not just file extension checks)
> - 60-second TTL signed download URLs
> - Private storage bucket for product files
>
> **Infrastructure Security:**
> - Rate limiting (120 req/min/IP)
> - Sentry error monitoring
> - Audit logging for all admin actions
> - Security contact via security.txt
>
> We built for the security standards that acquirers expect.
>
> **#Security #SaaS #FinTech #StartupAcquisition #CyberSecurity**

### Post 17: The Email Infrastructure
**Format:** Long-form / Feature highlight

> Transactional email is often an afterthought in early-stage platforms. On Keevan Store, it's a robust subsystem:
>
> - **DB-enqueued email queue:** Emails are queued via database triggers, not sent inline during requests
> - **Automatic retry:** Up to 3 retry attempts for failed sends
> - **Dual transport:** Resend API (primary) + Nodemailer/SMTP (fallback)
> - **Cron processing:** Daily batch processing at 06:00 UTC via Vercel Cron
> - **HTML templates:** Professional templates for welcome emails, purchase confirmations, order notifications, refund updates, and withdrawal processing
> - **Duplicate detection:** Prevents trigger storms from double-sending
>
> This means the platform can handle email at scale without performance degradation.
>
> For an acquirer, this is infrastructure that's already built, tested, and ready for scale.
>
> **#EmailInfrastructure #SaaS #CreatorEconomy #StartupAcquisition**

### Post 2

>

### Post 3
> The hardest technical challenge: payment verification.
>
> When a buyer pays on Pesapal, we need to verify the transaction three ways — client redirect, IPN webhook, and direct API query. Each path has edge cases. Each needs to be handled atomically.
>
> Getting this right was weeks of work. But it's the foundation of trust in a payment platform.
>
> #Engineering #Payments #SaaS #StartupJourney

### Post 4
> I chose Supabase over traditional cloud providers because:
> • Row-Level Security means every data access is authorized at the database level
> • PostgreSQL is battle-tested and reliable
> • The storage API handles file uploads securely
> • The auth system integrates seamlessly with Next.js SSR
>
> It was the right choice for a platform handling payments and user data.
>
> #TechStack #Supabase #SaaS #Engineering

### Post 5
